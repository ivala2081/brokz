Brokz was founded in 2025 with a specific thesis: institutional-grade trading infrastructure should not require a nine-figure budget or a multi-year development timeline to deploy. That thesis has shaped every decision we have made about what to build and how to build it.

This post is a transparent account of where we are, what we have built, and where we are going.

## What We Have Built

### MT4/MT5 Integration Stack

Our most mature product area is the MetaTrader integration stack. This includes:

**Bridge infrastructure.** We have built a modular bridge layer that connects MT5 server instances to multiple liquidity providers via FIX 4.2 and 4.4. The bridge handles price aggregation, order routing, and fill matching. It is designed to be deployed independently of the MT5 server, allowing bridge updates without server downtime.

**Manager API wrapper.** The raw MT5 Manager API is powerful but requires significant boilerplate to use correctly. We have built a typed wrapper that exposes the Manager API operations most relevant to brokerage operations — account management, position queries, trade history, and symbol configuration — with sensible error handling and connection management built in.

**Plugin framework.** Our plugin framework allows risk logic, reporting hooks, and custom business rules to be implemented as isolated MT5 server plugins that can be loaded, updated, and monitored independently.

### Risk Engine

Our risk engine is deployed as a service that integrates with the MT5 server and bridge layer. It handles:

- Pre-trade exposure checks against configurable position and notional limits
- Real-time P&L tracking across client accounts and the broker's own book
- Margin call and stop-out event processing
- Regulatory exposure reporting

The risk engine is designed to be configurable without code changes. Limit structures, margin tiers, and exposure rules are defined in a configuration layer that operations teams can modify through an admin interface.

### Analytics Infrastructure

We built a market data and analytics pipeline that captures tick data from MT5, normalizes it, and makes it available for:

- Historical backtesting by client-side quant teams
- Real-time spread monitoring and LP performance analysis
- Execution quality reporting (slippage, fill rate, latency)
- Client flow analysis

Data is stored in a time-series format optimized for high-frequency tick queries. The pipeline is designed to handle the data volumes generated by active multi-symbol deployments without degradation.

### Client Portal

We have built a white-label client portal that connects to the MT5 Manager API and provides:

- Account overview and real-time position display
- Deposit and withdrawal request workflow
- Trade history and statement generation
- Document management for KYC verification

The portal is built as a React application with a Node.js API layer. It is designed to be branded and customized for each brokerage client without requiring changes to the underlying codebase.

## Engineering Decisions We Made

A few decisions that shaped the architecture and are worth making explicit:

**Service decomposition from the start.** We made the deliberate choice not to build a monolithic system. The bridge, risk engine, analytics pipeline, and client portal are separate services with defined integration points. This added initial complexity but means each component can be scaled, updated, and replaced independently.

**No proprietary lock-in.** Every component we build is deployed into the client's own infrastructure. We do not operate a SaaS platform that clients depend on us to maintain. Clients own the code and can operate it independently. This is a principled choice that reflects our view of what a long-term technology partner relationship should look like.

**FIX over proprietary protocols.** For LP connectivity, we use FIX wherever it is available rather than proprietary APIs. FIX is the industry standard, and building against it means our bridge can connect to any FIX-compatible LP without custom integration work per provider.

**TypeScript for the operational layer.** The bridge and risk engine are implemented in TypeScript running on Node.js. This is not the conventional choice for latency-sensitive financial software — C++ or Go are more common. We made this choice because the latency profile of a retail/institutional hybrid brokerage (targeting milliseconds, not microseconds) is well within what Node.js can handle, and TypeScript's type system significantly reduces operational errors in complex financial logic.

## What Is Coming

### Algorithmic Trading Framework

We are building a framework for deploying algorithmic trading strategies that integrates directly with the MT5 execution stack. The framework handles strategy lifecycle management, position tracking, risk monitoring, and performance attribution. It is designed for institutional clients who want to run proprietary strategies through their MT5 infrastructure without building the surrounding management layer themselves.

Target release: Q4 2025.

### Enhanced Reporting Suite

The current reporting tools cover operational needs. We are building an expanded reporting layer focused on regulatory reporting — MiFID II transaction reporting, ESMA product governance reporting, and ASIC operational reporting — as well as enhanced client-facing account analysis tools.

Target release: Q1 2026.

### Multi-Custodian Settlement Layer

As more of our clients expand into multi-asset offerings, the settlement complexity increases. We are building a settlement abstraction layer that handles the different settlement mechanics of FX (T+2), equities (T+2 via custodian), and crypto (near real-time via exchange APIs) within a unified reconciliation framework.

This is the most complex item on the roadmap and the one we are spending the most engineering time on today.

Target release: H1 2026.
